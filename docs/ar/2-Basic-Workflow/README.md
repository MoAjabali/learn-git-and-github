# الوحدة الثانية: سير العمل الأساسي

في هذه الوحدة، سنتعلم الأوامر الأساسية لإدارة مشروعك محلياً وكيفية مزامنته مع منصة GitHub. سنركز ليس فقط على *ماذا* تكتب، ولكن على *كيف* تحرك هذه الأوامر عملك عبر دورة حياة Git.

## 1. تهيئة مشروع جديد (`git init`)
لبدء استخدام Git في أي مشروع، يجب عليك "تهيئته" أولاً. هذا الأمر ينشئ مجلداً مخفياً باسم `.git` في جذر مشروعك. هذا المجلد هو "عقل" Git—فهو يخزن كل التاريخ والإعدادات لهذا المستودع.

```bash
# 1. أنشئ مجلداً جديداً لمشروعك
mkdir my-cool-project

# 2. ادخل إلى المجلد
cd my-cool-project

# 3. اطلب من Git البدء في تتبع هذا المجلد
git init
```

> [!NOTE]
> تقوم بتشغيل `git init` **مرة واحدة فقط** لكل مشروع. إذا قمت بحذف مجلد `.git` المخفي، ستفقد كل سجل النسخ والتاريخ الخاص بمشروعك!

---

## 2. المراحل الثلاث: كيف "يفكر" Git؟
Git لا يقوم بحفظ كل شيء تلقائياً. يجب أن تكون منظماً في خطواتك. فكر في الأمر كعملية شحن بضائع:

1.  **مجلد العمل (Working Directory)**: المكان الذي تعدل فيه الملفات حالياً. التغييرات هنا تعتبر "غير متتبعة" أو "معدلة".
2.  **منطقة التحضير (Staging Area)**: "رصيف التحميل" حيث تختار ملفات معينة لتضمينها في الشحنة التالية.
3.  **المستودع المحلي (Commit)**: المكان الذي تغلق فيه الصندوق وتضع عليه ملصقاً (رسالة الـ commit). الآن تم حفظ التغيير بأمان في تاريخك.

### سير العمل البرمجي (The Command Workflow)
عندما تقوم بالبرمجة، ستكرر هذه الدورة مئات المرات:

#### أ. التحقق من الحالة الراهنة
قبل القيام بأي خطوة، تحقق دائماً مما يراه Git:
```bash
git status
```
*لماذا؟* يخبرك هذا الأمر بالملفات المعدلة، والملفات الجديدة، وما الذي تم نقله بالفعل إلى "رصيف التحميل" (Staged).

#### ب. نقل الملفات إلى رصيف التحميل (Staging)
إذا انتهيت من مهمة محددة (مثل إضافة شعار للموقع)، أضف تلك الملفات فقط:
```bash
# إضافة ملف محدد
git add images/logo.png

# أو إضافة جميع التغييرات في المجلد الحالي (كن حذراً!)
git add .
```

#### ج. إغلاق الصندوق والحفظ (Committing)
الآن، احفظ هذه التغييرات بشكل دائم مع وصف لما قمت به:
```bash
git commit -m "feat: add company logo to header"
```
*لماذا؟* بدون عمل commit، تغييراتك مجرد "مسودات". الـ commit ينشئ لقطة دائمة يمكنك العودة إليها في أي وقت مستقبلاً.

---

## 3. كتابة رسائل Commit احترافية
رسالة الـ Commit هي ملاحظة لنفسك في المستقبل ولزملائك. في البيئات الاحترافية، نتبع نمط **Conventional Commits** (سنتوسع في هذا في الوحدة الخامسة).

**القاعدة الذهبية**: استخدم **صيغة الأمر**. تخيل أنك تعطي أمراً للكود ليقوم بشيء ما.
- ✅ `feat: add user login` (تخبر Git: "أضف نظام تسجيل الدخول")
- ❌ `Added user login` (وصف لما فعلته بالأمس - غير مفضل)
- ❌ `User login stuff` (كلام عام وغير محدد)

---

## 4. الاتصال بـ GitHub (المستودعات البعيدة)
يجب أن يعرف "عقل" Git المحلي مكان وجود "العقل السحابي" (GitHub). نسمي هذه الروابط **Remotes**.

```bash
# أضف رابطاً بعيداً وقم بتسميته 'origin' (الاسم القياسي)
git remote add origin https://github.com/username/repo-name.git

# تحقق من أن الرابط صحيح
git remote -v
```

### الدفع (Push) والسحب (Pull): التزامن
- **`git push origin main`**: يأخذ الـ commits المحلية و"يرفعها" إلى فرع `main` على GitHub.
- **`git pull origin main`**: "يُحمل" آخر التغييرات من GitHub ويدمجها في عملك المحلي.

> [!TIP]
> **أفضل ممارسة**: قم دائماً بعمل `git pull` قبل أن تبدأ العمل لتتأكد أن لديك أحدث نسخة من الكود وتتجنب التعارضات!

## ماذا بعد؟
الآن وقد تعلمت كيف تحفظ وتشارك عملك، حان الوقت لتعلم كيفية العمل على ميزات مختلفة في نفس الوقت باستخدام **الفروع (Branches)**!
